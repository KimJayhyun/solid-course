# SOLID 객체 지향 설계 원칙

이 저장소는 SOLID 객체 지향 설계의 5가지 기본 원칙을 Java 예제 코드와 함께 설명합니다. 각 원칙은 응집도 높고 결합도 낮은, 유연하고 유지보수하기 쉬운 소프트웨어를 만드는 데 도움을 줍니다.

---

## 1. 단일 책임 원칙 (Single Responsibility Principle, SRP)

> **"클래스를 변경해야 하는 이유는 단 하나여야 한다."**

클래스는 단 하나의 책임(기능)만 가져야 합니다. 책임이 분리되면 코드의 이해와 유지보수가 쉬워지고, 변경의 영향 범위가 줄어듭니다.

- **[자세히 보기: 단일 책임 원칙](./docs/01.%20Single%20Responsibility%20Principle.md)**

## 2. 개방-폐쇄 원칙 (Open-Closed Principle, OCP)

> **"소프트웨어 엔티티(클래스, 모듈, 함수 등)는 확장에는 열려 있어야 하고, 수정에는 닫혀 있어야 한다."**

기능을 추가하거나 변경할 때 기존 코드를 수정하는 대신, 상속이나 구성을 통해 기능을 확장할 수 있어야 합니다. 이를 통해 기존 코드의 안정성을 유지할 수 있습니다.

- **[자세히 보기: 개방-폐쇄 원칙](./docs/02.%20Open%20Closed%20Principle.md)**

## 3. 리스코프 치환 원칙 (Liskov Substitution Principle, LSP)

> **"부모 클래스 객체를 자식 클래스 객체로 치환해도 프로그램의 정확성이 깨져서는 안 된다."**

자식 클래스는 부모 클래스의 동작 규약을 깨뜨리지 않아야 합니다. 즉, 부모 클래스가 사용되는 곳이라면 어디든 자식 클래스로 대체할 수 있어야 합니다.

- **[자세히 보기: 리스코프 치환 원칙](./docs/03.%20Liskov%20Substitution%20Principle.md)**

## 4. 인터페이스 분리 원칙 (Interface Segregation Principle, ISP)

> **"클라이언트는 자신이 사용하지 않는 메서드에 의존하도록 강요받아서는 안 된다."**

하나의 거대한 인터페이스보다는, 여러 개의 작고 구체적인 인터페이스로 분리하는 것이 좋습니다. 이를 통해 클라이언트는 자신에게 필요한 기능만 사용하게 되어 불필요한 종속성이 줄어듭니다.

- **[자세히 보기: 인터페이스 분리 원칙](./docs/04.%20Interface%20Segregation%20Principle.md)**

## 5. 의존관계 역전 원칙 (Dependency Inversion Principle, DIP)

> **"상위 모듈은 하위 모듈에 의존해서는 안 된다. 둘 다 추상화에 의존해야 한다."**
>
> **"추상화는 세부사항에 의존해서는 안 된다. 세부사항이 추상화에 의존해야 한다."**

구체적인 구현 클래스에 직접 의존하는 대신, 추상화(인터페이스)에 의존해야 합니다. 의존성 주입(Dependency Injection)을 통해 모듈 간의 결합도를 낮추고 유연성을 높일 수 있습니다.

- **[자세히 보기: 의존관계 역전 원칙](./docs/05.%20Dependency%20Inversion%20Principle.md)**
